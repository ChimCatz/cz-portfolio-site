Call Attribution

VOIP Call Conversion Tool

Case Study View GitHub Repository

This project came from a repetitive task that was quietly draining my time as a data analyst. I regularly had to cross-match call logs from a VOIP system with CRM lead data to identify which interactions actually converted. On paper that sounds simple, but the real work started long before the matching itself. I first had to clean both datasets manually, and that preparation alone usually took close to an hour for each run.

The VoIP call logs took around 30 minutes to prepare because formats were inconsistent, columns needed to be aligned, and phone numbers often had to be cleaned by hand. The CRM export added another 20 to 30 minutes because contact numbers were scattered across multiple fields and the structure was not always consistent. So instead of trying to speed up the cleanup work manually, I decided to remove it from the workflow entirely.

What came out of that was the VOIP Call Conversion Tool, a small desktop utility built to take raw call data and CRM exports, standardize them, and produce a clean conversion report in seconds instead of rebuilding the same process manually every time.

I built a desktop tool that lets me import raw CSV files from both sources and handle the cleanup, normalization, matching, and reporting in the backend. One of the first problems I solved was format inconsistency. Different VoIP exports follow different structures, so I added a detection layer that identifies the format based on column patterns and routes the file through the correct processing logic. Once detected, the tool normalizes the data into a standard structure so dates, durations, and call directions all follow the same rules.

Phone numbers were another major source of unreliable matching. They came in different formats with spaces, symbols, and country codes, which caused frequent mismatches. To fix that, I added a cleaning step that strips everything except digits so numbers are normalized before matching begins. That made the matching logic much more dependable across both data sources.

On the CRM side, the issue was fragmented contact data. Phone numbers were not stored in a single field. They could appear in mobile, direct line, or telephone columns, depending on the record. Instead of picking one field and hoping for the best, the tool extracts every possible number per contact and restructures them into a normalized dataset. That increases matching coverage without relying on guesswork.

For the actual conversion logic, I avoided relying only on exact matches because real-world data is rarely that clean. The system uses a layered approach: exact match on the full number first, then fallback checks using the last 8 digits and last 7 digits. That gives the tool a better chance of matching slightly inconsistent data while still keeping the process controlled. Matching is also performed across multiple call fields rather than a single column, which improves overall coverage.

I also added simple business rules so the results reflect meaningful activity rather than noise. Not every call should count toward conversion performance, so the tool filters for eligible calls using criteria like call direction and minimum connection time. That way the final metrics are based on valid interactions instead of inflated raw volume.

The interface is intentionally simple. The workflow is reduced to four steps: import call logs, import CRM data, run conversion, and export results. Behind that simple sequence, the tool automatically calculates metrics like total calls, unique contacts, converted calls, effectiveness rate, and coverage rate, so there is no need for manual computation anymore.

The impact was immediate. What used to take around 60 minutes per dataset now takes under 10 to 15 seconds from import to export. More importantly, the process is now repeatable and consistent. I no longer depend on manual cleanup, and I no longer have to worry about avoidable matching errors every time a new dataset comes in.
